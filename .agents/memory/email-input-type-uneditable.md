---
name: Auth email inputs blocked by native browser autofill
description: Recurring bug — native browser autofill/credential-suggestion UI recognizes email fields by label/placeholder/form-shape (not just type/autoComplete attributes) and captures the keyboard from the first keystroke; attribute-only mitigation is confirmed insufficient, readOnly-until-focus is the reinforced fix.
---

**Rule:** ALL email inputs site-wide — not just auth pages — must minimize autofill-detection signals AND tolerate autofill writes that bypass React:
- No `type="email"`, no `autoComplete="email"`/`inputMode="email"` (use `autoComplete="off"`, `inputMode="text"`), DOM `name` must not contain "email" (override after the RHF `{...field}` spread — RHF Controller is ref-based), `data-lpignore` + `data-1p-ignore`.
- Placeholder must not contain `@` or an email-shaped string (e.g. `nombre@email.com`) — browsers' autofill heuristics scan placeholder text too, not just attributes. Use plain text like "Introduce tu correo".
- No `autoFocus` on the field — auto-focusing on mount is exactly the moment the autofill engine hooks the field during hydration.
- **readOnly until first focus** (the reinforced layer, added 2026-07-11): render the input `readOnly={locked}` with `locked` initialized `true`, and `onFocus={() => setLocked(false)}`. This stops the browser's autofill/credential-manager engine from recognizing and attaching to the field at initial render — it only becomes editable once the user has deliberately focused it. Order matters for RHF fields: put `readOnly`/`onFocus` *before* `{...field}` in JSX so `field`'s own props (value/onChange/onBlur/name/ref — it has no `onFocus`) don't get clobbered.
- Custom `onBlur`: if the DOM value differs from the controlled value, sync it back into state before finalizing blur — captures native autofill that skips React's onChange. For RHF: `if (e.target.value !== field.value) field.onChange(e.target.value); field.onBlur();`. For plain `useState` forms: `if (e.target.value !== form.email) set({ email: e.target.value })`.

**Why:** Root cause is broader than first diagnosed. Original 2026-07 diagnosis ("native autofill widget captures focus on click") led to an attribute-scrubbing recipe (no `type=email`/`autoComplete=email`/`inputMode=email`) applied to `login.tsx` twice (Thu commits) and confirmed by the user as fixed at the time. It then recurred on `team.tsx`'s create/edit-user dialogs (different screen, same anti-pattern, never touched by the login.tsx fixes) — fixed 2026-07-11. **Then the user retested `/register` in production on 2026-07-11 across Safari, Chrome desktop, and mobile, and found the email field still 100% blocked from the very first keystroke on every one of them** — despite the attribute recipe already being live there. A block that's total and identical across Chromium and WebKit engines on both desktop and mobile is inconsistent with a per-browser autofill-dropdown quirk (those normally still let you type around the suggestion); it's consistent with the browser/OS-level credential-suggestion UI (Chrome's autofill, Safari/iCloud Keychain) recognizing the *whole form shape* (name + email + password + confirm-password fields, label text "Email", placeholder containing "@"/"email.com") and hooking the field before or regardless of individual DOM attributes, especially for real user profiles with saved credentials (a sandboxed/clean test browser without saved autofill data reproduces no block at all — confirmed by testing `/register` live, which typed successfully on the first try in a clean session). This explains why the same attribute-only recipe worked for the user once (transient/env-dependent) and then stopped working: it was never fully eliminating the trigger, just reducing the odds.

**Confirmed fixed (2026-07-11):** user retested `/register` in production after the readOnly-until-focus reinforcement, across Chrome desktop, Safari desktop (with Keychain/saved credentials), and mobile — works on all three. This is the strongest validation this bug has had; if it recurs again, it's likely a genuinely new trigger (e.g. a different form shape or a new field), not the same one — re-verify from scratch rather than assuming the existing recipe just needs reapplying.

**How to apply:** Any email field anywhere in the app follows the full recipe above (attributes + placeholder wording + no autoFocus + readOnly-until-focus). Check by grepping for `inputMode="email"\|autoComplete="email"\|type="email"` project-wide, not just on auth pages. Fixed with the reinforced (readOnly-until-focus) recipe as of 2026-07-11: `login.tsx` (login + register email fields, `autoFocus` also removed from the login one), `team.tsx` (create + edit user dialogs), `trip-travelers-tab.tsx` (invite-traveler email field). If it recurs even after this reinforced recipe, the next thing to try is temporarily renaming the visible label away from "Email" (UX cost) or moving the email field to its own isolated `<form>`/step, since forms with an adjacent password field are the strongest trigger for browsers' credential-manager heuristics. Longer-term fix: extract a single shared `<AutofillSafeEmailInput>` component so the mitigation only has to be applied and updated once instead of per-screen.
