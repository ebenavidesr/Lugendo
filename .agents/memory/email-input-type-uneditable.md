---
name: Auth email inputs blocked by native browser autofill
description: Recurring bug — native email/address autofill steals focus on any email input site-wide (not just login/register) and blocks typing until reload; mitigation recipe (not 100% eliminable) must be applied to EVERY email field independently, since each screen has its own hand-rolled input.
---

**Rule:** ALL email inputs site-wide — not just auth pages — must minimize autofill-detection signals AND tolerate autofill writes that bypass React:
- No `type="email"`, no `autoComplete="email"`/`inputMode="email"` (use `autoComplete="off"`, `inputMode="text"` — `inputMode="email"` alone is a strong mobile-autofill signal even without `autoComplete="email"`), DOM `name` must not contain "email" (override after the RHF `{...field}` spread — RHF Controller is ref-based), `data-lpignore` + `data-1p-ignore`.
- Custom `onBlur`: if the DOM value differs from the controlled value, sync it back into state before finalizing blur — captures native autofill that skips React's onChange. For RHF: `if (e.target.value !== field.value) field.onChange(e.target.value); field.onBlur();`. For plain `useState` forms: `if (e.target.value !== form.email) set({ email: e.target.value })`.

**Why:** User-confirmed root cause (2026-07): the browser's native email/address autocomplete widget (visible as a `chrome-extension://` frame in devtools) captures focus on click/refocus and the field stops responding to the keyboard until reload. Reproduced in the published app, new tabs, and incognito — not an iframe or app-logic issue. Known React-controlled-form vs browser-autofill conflict; realistic goal is reducing frequency, not eliminating it. Regressed twice before on `login.tsx` when `type="email"`/`autoComplete="email"` were re-added "for autofill" — **and then recurred a third time (2026-07-11) on a completely different screen**: `team.tsx`'s create/edit-user dialogs (alta de usuario) had their own hand-rolled `Input` with `inputMode="email"` + `autoComplete="email"` that was never touched by the two earlier login.tsx fixes, because each screen implements its own email field rather than sharing one component.

**How to apply:** Any email field anywhere in the app follows the recipe above — check by grepping for `inputMode="email"\|autoComplete="email"\|type="email"` project-wide, not just on auth pages. Fixed so far: `login.tsx` (both fields), `team.tsx` (create + edit user dialogs), `trip-travelers-tab.tsx` (invite-traveler email field) — all as of 2026-07-11. If incidents persist even after the recipe is applied, next hardening step: also re-sync the DOM value on `onFocus`, and consider changing the placeholder (contains "@", another heuristic signal). Longer-term fix: extract a single shared `<AutofillSafeEmailInput>` component so the mitigation only has to be applied once instead of per-screen.
