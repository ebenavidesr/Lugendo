---
name: Auth email inputs blocked by native browser autofill
description: Recurring bug family ‚Äî native browser credential-manager UI (Chrome autofill, Safari/iCloud Keychain, mobile) recognizes email AND password fields by form shape/attributes (not just type/autoComplete) and captures the keyboard from the first keystroke, sometimes blocking paste too; attribute-only mitigation is confirmed insufficient, readOnly-until-focus is the reinforced fix ‚Äî applies to any credential-shaped field (email, password, confirm-password), not just email.
---

**Rule:** ALL credential-shaped inputs site-wide (email AND password fields) ‚Äî not just auth pages ‚Äî must minimize autofill-detection signals AND tolerate autofill writes that bypass React:
- No `type="email"`, no `autoComplete="email"`/`inputMode="email"` (use `autoComplete="off"`, `inputMode="text"`) for email fields. Password fields keep `autoComplete="current-password"`/`"new-password"` (needed for legitimate password-manager UX ‚Äî the goal there is not blocking the manager, just not letting it block the user).
- DOM `name` must not contain "email" (override after the RHF `{...field}` spread ‚Äî RHF Controller is ref-based), `data-lpignore` + `data-1p-ignore`.
- Placeholder must not contain `@` or an email-shaped string (e.g. `nombre@email.com`) ‚Äî browsers' autofill heuristics scan placeholder text too, not just attributes. Use plain text like "Introduce tu correo".
- No `autoFocus` on the field ‚Äî auto-focusing on mount is exactly the moment the autofill engine hooks the field during hydration.
- **readOnly until first focus** (the reinforced layer): render the input `readOnly={locked}` with `locked` initialized `true`, and `onFocus={() => setLocked(false)}`. This stops the browser's autofill/credential-manager engine from recognizing and attaching to the field at initial render ‚Äî it only becomes editable once the user has deliberately focused it. Order matters for RHF fields: put `readOnly`/`onFocus` *before* `{...field}` in JSX so `field`'s own props (value/onChange/onBlur/name/ref ‚Äî it has no `onFocus`) don't get clobbered.
- Custom `onBlur`: if the DOM value differs from the controlled value, sync it back into state before finalizing blur ‚Äî captures native autofill that skips React's onChange. For RHF: `if (e.target.value !== field.value) field.onChange(e.target.value); field.onBlur();`. For plain `useState` forms: `if (e.target.value !== form.email) set({ email: e.target.value })`.

**Why:** Root cause is broader than first diagnosed, twice. Timeline:
1. Original 2026-07 diagnosis ("native autofill widget captures focus on click") ‚Üí attribute-scrubbing recipe (no `type=email`/`autoComplete=email`/`inputMode=email`) applied to `login.tsx` email field twice (Thu commits), user confirmed fixed at the time.
2. Recurred on `team.tsx`'s create/edit-user dialogs (different screen, same anti-pattern, never touched by the login.tsx fixes) ‚Äî fixed 2026-07-11.
3. User retested `/register` email field in production on 2026-07-11 across Safari, Chrome desktop, and mobile ‚Äî still 100% blocked from the first keystroke on every one, despite the attribute recipe. A total, engine-agnostic block like that isn't a per-browser autofill-dropdown quirk (those normally still let you type around the suggestion) ‚Äî it's the browser/OS-level credential-suggestion UI recognizing the whole form shape and hooking the field regardless of attributes, especially with saved credentials present (a clean/sandboxed test browser reproduces no block at all, which is why testing in an automation browser without saved autofill data gives false negatives). **Fix: readOnly-until-focus recipe, added to the email fields. User confirmed this worked across Chrome desktop, Safari desktop (with Keychain), and mobile.**
4. **Same day, immediately after: user reports the *password* field on `/login` now has the identical symptom** ‚Äî 100% blocked from the first keystroke, on mobile and desktop, Safari and Chrome ‚Äî plus paste is also blocked (ruled out any in-app `onPaste`/`onKeyDown` handler; grepped the whole frontend, none exist ‚Äî confirms this is native browser/OS behavior, not app code). The password manager doesn't even visibly prompt; it just silently eats all keyboard and clipboard input. This is the same bug class as the email one, just on a different field type that hadn't been hardened yet. Applied the same readOnly-until-focus recipe to `login.tsx`'s login password field, and preemptively to the register password + confirm-password fields (same shape, not yet reported broken, but no reason to wait for a third round-trip).

**Lesson ‚Äî don't declare this bug class "fixed," only "fixed for the fields hardened so far":** every credential-shaped field (email, password, confirm-password) on every form is an independent target for the browser's heuristics; fixing one field does not fix siblings on the same form. When a "same error" report comes in again, check *which specific field* first (don't assume it's the same field as last time ‚Äî twice now it's been a different one: team.tsx's email, then login's password) ‚Äî grep for `type="password"` and `type="email"`/email-shaped inputs across the frontend and check each one has the full recipe.

**How to apply:** Any email or password field anywhere in the app follows the full recipe above (attributes + placeholder wording + no autoFocus + readOnly-until-focus). Grep for `inputMode="email"\|autoComplete="email"\|type="email"` and separately check every `type="password"` input, project-wide, not just on auth pages. Hardened as of 2026-07-11: `login.tsx` (login email + password, register email + password + confirm-password ‚Äî `autoFocus` also removed from the login email field), `team.tsx` (create + edit user dialogs, email only ‚Äî **password fields there still unhardened, check if reported**), `trip-travelers-tab.tsx` (invite-traveler email field, no password field on that form). If it recurs even after this reinforced recipe on a given field, the next thing to try is temporarily renaming the visible label away from "Email"/"Contrase√±a" (UX cost) or moving the field to its own isolated `<form>`/step, since forms with adjacent email+password fields are the strongest trigger for browsers' credential-manager heuristics. Longer-term fix: extract shared `<AutofillSafeEmailInput>`/`<AutofillSafePasswordInput>` components so the mitigation only has to be applied and updated once instead of per-field-per-screen.
